# 仕様決定事項

このドキュメントは、実装前に決めておくべき仕様上の論点を管理します。

## 決定済みの仕様方針

### 1. 蔵書管理に集中する

このアプリは、読書記録や欲しい本リストではなく、所有している本を管理するための蔵書管理アプリとして設計します。

決定済み:

- 感想、レビュー、読書履歴はMVPに含めない
- 欲しい本リストはMVPに含めない
- 読了日、読書ステータスはMVPに含めない
- 管理メモは感想ではなく、破損、付属品、棚卸し補足など管理用途に限定する

### 2. 本のバーコードと管理用バーコードを分ける

本に印刷されているバーコードと、利用者が独自に貼付する管理用バーコードを別項目として扱います。

決定済み:

- 本のバーコードは書誌情報取得と検索に使う
- 管理用バーコードは蔵書の一意識別と検索に使う
- 管理用バーコードは外部API照会には使わない
- 管理用バーコードは入力された場合、一意である必要がある

### 3. 外部APIで書誌情報を取得する

本のバーコードを読み込んだとき、外部APIから書誌情報を取得できるようにします。

決定済み:

- MVPに外部API連携を含める
- 取得できた書誌情報は登録フォームに自動入力する
- 外部APIで見つからない場合も手入力登録できる
- 外部APIの失敗で本登録全体をブロックしない
- APIキー必須の外部APIには依存しない
- MVPの外部APIはOpen Library APIを利用する
- Google Books APIはMVP必須にせず、将来の任意プロバイダー候補に留める
- ISBNまたはISBNとして解釈できる本のバーコードは、Open Library ISBN APIで完全一致照会する
- ISBN APIで取得できた場合は1件の書誌情報として扱い、候補選択UIは表示しない
- ISBN APIで取得できない場合はSearch APIへフォールバックせず、手入力登録に進む
- MVPでは外部APIによる候補選択UIは実装しない
- Open Library APIの利用時は `User-Agent` と連絡先を設定する
- 同一ISBNや同一本バーコードへの照会結果はキャッシュする
- 外部APIのレート制限を超えないよう、アプリ側で連続照会を制御する

### 4. 保管場所は事前登録タグとして扱う

保管場所は自由入力ではなく、事前に登録した保管場所タグから選択します。

決定済み:

- MVPに保管場所タグ管理を含める
- 本の登録、編集では保管場所タグを選択する
- 既に本に紐づく保管場所タグは削除せず、無効化する
- MVPでは1冊の本に設定できる保管場所タグは1つとする

### 5. 分類はAPI情報を優先し、不足時は分類タグで補う

本の分類情報は、Open Library APIから取得できるsubjectを優先します。APIから分類情報を取得できない場合や、運用上追加したい分類がある場合は、分類タグとして手動登録します。

決定済み:

- Open Library APIのsubjectは分類タグ候補として使う
- ユーザーが確定したsubjectのみ分類タグとして保存する
- APIに分類情報がない場合は分類タグを手動登録する
- 分類タグは保管場所タグとは別の概念として扱う

### 6. Export/Importを提供する

自宅や研究室でセルフホストする前提のため、データ移行とバックアップに使えるExport/Importを提供します。

決定済み:

- MVPにJSON Export/Importを含める
- Export対象は本、保管場所タグ、分類タグ、本と分類タグの関連
- Import時の重複はWindowsでフォルダをコピーしたときのように、個別選択、すべて上書き、すべて無視を選べるようにする
- MVPでは全データ置換Importは実装せず、追加Importと選択的な上書きを基本とする
- CSV Export/Importは必要になった時点で検討する

## 補足仕様

### 1. 著者の複数入力

複数著者をどのように入力、表示、検索するかを決めます。

推奨案:

- MVPでは1つの文字列として保持する
- 入力時はカンマ区切りを許可する
- 検索は著者文字列への部分一致で行う

決定状況:

- 暫定決定済み

### 2. バーコード読み取り方法

Webアプリでどの読み取り方法をMVP対象にするかを決めます。

推奨案:

- MVPではバーコードリーダーがキーボード入力として値を送る方式を前提にする
- カメラ読み取りはPhase 2以降に回す

決定状況:

- 暫定決定済み

### 3. Importの重複判定

Import時の重複判定ルールを決めます。

決定済み:

- 管理用バーコードが一致する本は重複として扱う
- 保管場所タグ名が一致する保管場所タグは重複として扱う
- 分類タグ名が一致する分類タグは重複として扱う
- ISBNのみの一致は重複確定ではなく警告として扱う
- 重複項目はImportプレビューで表示し、個別に上書きまたは無視を選択できる
- 重複項目に対して `すべて上書き` と `すべて無視` を選択できる
- 初期選択は `無視` とする
- JSON形式の不正、必須項目不足、関連データの参照不整合は、ユーザー選択では解決できないエラーとして扱う

決定状況:

- 決定済み

### 4. 保管場所タグの初期データ

初期状態で保管場所タグを用意するかを決めます。

推奨案:

- 初期データはなし
- ユーザーが最初に保管場所タグを登録する
- 開発、テスト用のサンプルデータは本番データとは分ける

決定状況:

- 決定済み
- 初期データは用意しない

### 5. 空状態の振る舞い

本や保管場所タグが未登録のとき、または検索結果が0件のときの表示を決めます。

推奨案:

- 本が未登録のときは本登録画面への導線を表示する
- 保管場所タグが未登録のときは保管場所登録画面への導線を表示する
- 検索結果0件時は検索条件の解除導線を表示する

決定状況:

- 決定済み

## 先に決めなくてよい仕様

以下はMVP実装中に必要になってから決めます。

- 表紙画像の扱い
- カメラによるバーコード読み取り
- CSV Import/Export
- 貸出管理
- 複数ユーザー、認証、権限
- 棚卸し専用ワークフロー

## 実装前の最小決定セット

仕様面の主要論点は決定済みです。実装時に詳細化が必要になった場合は、実装変更と同じPull RequestでDocsを更新します。
