# 実装前決定事項

このドキュメントは、実装に入る前に決めておくべき事項を管理します。

## すぐ決めるべき事項

### 1. 技術スタック

実装開始前に、フロントエンド、バックエンド、データベース、ORM、テストの基本構成を決めます。

決定済み:

- Frontend: React + TypeScript + Vite
- Backend: Node.js + TypeScript + Fastify
- Database: SQLite
- ORM: Drizzle ORM
- API: REST
- Test: フロントエンドはVitest、バックエンドはNode.js標準テストランナーまたはVitestを候補にする

方針:

- フロントエンドとバックエンドの型定義を揃えやすいよう、TypeScript中心で構成する
- セルフホストしやすいよう、Docker Composeを初期から用意する

### 1.1 開発環境管理

ローカル開発に必要なランタイム、ツール、タスク実行はmiseで管理します。

決定済み:

- 環境作成にはmiseを使う
- Node.js、Pythonなどのバージョンはリポジトリ内のmise設定で固定する
- セットアップ、開発サーバー起動、テストなどの共通タスクも可能な範囲でmiseタスクとして定義する

方針:

- `mise.toml` または `.mise.toml` をリポジトリルートに配置する
- 実装技術が確定した時点で必要なツールバージョンをDocsへ追記する
- Dockerを導入する場合も、ローカルの補助タスク実行はmiseに寄せる

### 2. 初期デプロイ方針

ローカル専用で始めるか、セルフホストしやすい構成にするかを決めます。

決定済み:

- 自宅や研究室のセルフホスト環境で動かせる構成を前提にする
- MVPはローカル開発で完結させつつ、あとからセルフホストしやすいよう設定値を環境変数化する
- Docker Composeを標準のセルフホスト起動方法とする
- SQLiteデータベースファイルはDocker volumeまたはホストディレクトリへマウントする
- SQLiteデータベースファイル自体をバックアップ対象とする
- 外部書誌APIはAPIキー必須の仕組みに依存しない
- Open Library API向けのアプリ名、連絡先は環境変数で設定できるようにする

### 2.1 カメラ読み取り連携

初期フロントエンドはWebアプリケーションとして実装します。HTTP環境ではWebブラウザのカメラ利用に制約があるため、将来的にフロントエンド部分をAndroidアプリケーションへ置き換え、または追加できるように、フロントエンドとバックエンドを分離します。

決定済み:

- カメラ読み取りは優先事項として扱う
- バーコード読み取り機能は、読み取った文字列をAPIへ渡す入力元として扱う
- バックエンドAPIはWebフロントエンド固有の状態に依存しないJSON APIにする
- Androidフロントエンドを作る場合も、Backend APIを再利用する
- AndroidフロントエンドはMVP対象外とする

### 3. IDの形式

`books.id` を整数連番にするかUUIDにするかを決めます。

決定済み:

- Export/Importや外部クライアント連携での衝突を避けるため、IDはUUIDとする

### 4. ISBNの扱い

ISBNの正規化と重複時の扱いを決めます。

推奨初期案:

- ハイフンあり、なしの入力を許可する
- 保存時は数字と `X` のみに正規化する
- 重複登録は許可し、画面上で警告する方針に留める

### 5. 一覧の並び順

本一覧の初期表示順を決めます。

推奨初期案:

- 初期表示は `updated_at` の降順
- 検索や絞り込みを行っても同じ並び順を維持する

### 6. 削除方針

MVPでは物理削除とします。削除履歴や復元が必要になった時点で論理削除を検討します。

決定済み:

- MVPは物理削除

### 7. 認証の扱い

MVPでは自宅や研究室のセルフホスト環境で動く単一インスタンス利用を前提にし、ログイン機能は実装しません。

決定済み:

- MVPでは認証なし
- 複数ユーザー対応はPhase 2以降

### 8. UIの基本方針

一覧画面をテーブル中心にするか、カード中心にするかを決めます。

推奨初期案:

- PCではテーブル中心
- スマートフォンではカード表示に切り替える
- 登録、編集フォームは同じ入力部品を再利用する

### 9. 開発単位

実装はGitHub Issue単位で進めます。

推奨初期案:

- `documentation`: Docs更新
- `enhancement`: 機能実装
- `good first issue`: 小さなUIやDocs改善

## 実装前の最小決定セット

最低限必要な技術方針は決定済みです。

決定済みの前提:

- 環境作成にはmiseを使う
- フロントエンドはReact + TypeScript + Vite
- バックエンドはNode.js + TypeScript + Fastify
- ORMはDrizzle ORM
- DBはSQLite
- IDはUUID
- Docker Composeを初期から用意する

その他は推奨初期案のまま進め、必要になった時点でDocsを更新します。
