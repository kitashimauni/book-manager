# 実装前決定事項

このドキュメントは、実装に入る前に決めておくべき事項を管理します。

## すぐ決めるべき事項

### 1. 技術スタック

実装開始前に、フロントエンド、バックエンド、データベース、ORM、テストの基本構成を決めます。

推奨初期案:

- Frontend: React + TypeScript + Vite
- Backend: Node.js + TypeScript + Express または FastAPI
- Database: SQLite
- API: REST
- Test: フロントエンドはVitest、バックエンドは選定技術に合わせて導入

未決定:

- バックエンドを Node.js 系にするか Python/FastAPI 系にするか
- ORMまたはクエリビルダーを使うか
- Dockerを最初から使うか

### 2. 初期デプロイ方針

ローカル専用で始めるか、早期に外部公開できる構成にするかを決めます。

推奨初期案:

- MVPはローカル開発で完結させる
- デプロイはPhase 1の終盤またはPhase 2で検討する

### 3. IDの形式

`books.id` を整数連番にするかUUIDにするかを決めます。

推奨初期案:

- SQLiteで始める場合はINTEGERの自動採番
- 将来の外部連携やデータ移行を重視する場合はUUID

### 4. ISBNの扱い

ISBNの正規化と重複時の扱いを決めます。

推奨初期案:

- ハイフンあり、なしの入力を許可する
- 保存時は数字と `X` のみに正規化する
- 重複登録は許可し、画面上で警告する方針に留める

### 5. 一覧の並び順

本一覧の初期表示順を決めます。

推奨初期案:

- 初期表示は `updated_at` の降順
- 検索や絞り込みを行っても同じ並び順を維持する

### 6. 削除方針

MVPでは物理削除とします。削除履歴や復元が必要になった時点で論理削除を検討します。

決定済み:

- MVPは物理削除

### 7. 認証の扱い

MVPでは単一ユーザー利用を前提にし、ログイン機能は実装しません。

決定済み:

- MVPでは認証なし
- 複数ユーザー対応はPhase 2以降

### 8. UIの基本方針

一覧画面をテーブル中心にするか、カード中心にするかを決めます。

推奨初期案:

- PCではテーブル中心
- スマートフォンではカード表示に切り替える
- 登録、編集フォームは同じ入力部品を再利用する

### 9. 開発単位

実装はGitHub Issue単位で進めます。

推奨初期案:

- `documentation`: Docs更新
- `enhancement`: 機能実装
- `good first issue`: 小さなUIやDocs改善

## 実装前の最小決定セット

最低限、次の3点を決めれば実装に入れます。

1. フロントエンドとバックエンドの技術スタック
2. `books.id` の形式
3. Dockerを初期導入するかどうか

その他は推奨初期案のまま進め、必要になった時点でDocsを更新します。
